## 🗂️ PART 2 — DATABASE ARCHITECTURE (ERD-Level)

### 2.1 — Multi-Tenancy Strategy (MOST CRITICAL DECISION)

You need to decide this **before** any table design.

**Option A — Single DB, `campus_id` everywhere**
Every table has a `campus_id` foreign key. Simpler but risky if not enforced.

**Option B — Separate DB per campus**
Cleanest isolation. Complex migrations.

**Recommended: Option A with Global Scopes**
Use Laravel's Global Scopes to automatically inject `campus_id` on every query. This means one codebase serves all campuses without data leaks.

## 📦 PART 4 — MODULE-BY-MODULE DEEP DIVE

### MODULE 1 — Admission Management

**Flow:**
```
Inquiry → Application Form → Document Upload → Entry Test → 
Merit List → Offer Letter → Fee Payment → Enrollment → 
Class Assignment → ID Card Generation
```

**Edge Cases to Handle:**
- Duplicate B-Form detection (same child, two applications)
- Waitlist management when class is full
- Partial document submission — allow saving draft applications
- Re-admission of previous students (pull old record)
- Transfer students — different checklist

**Key Features:**
- Admission form with configurable fields per class
- Auto-generate registration number on approval
- Email/SMS on each stage change
- Document checklist with status per document
- Merit list auto-calculation based on entry test marks

---

### MODULE 2 — Student Management

**Student Status Lifecycle:**
```
Applicant → Enrolled → Active → [Promoted / Detained / Transferred / Left / Expelled] → Alumni
```

**Critical Data Points:**
- B-Form no. must be unique per campus
- Two contacts minimum (father + mother/guardian)
- Emergency contact mandatory
- Photo requirements (passport size, recent)
- Blood group (for emergencies)

**Promotion Logic (Annual Process):**
- Bulk promotion: pass students move to next class
- Detained students stay in same class
- New academic year auto-created from template
- Roll numbers re-assigned after promotion
- Fee structures copied to new year with price-edit option

---

### MODULE 3 — Timetable System

**Clash Detection Rules:**
1. A teacher cannot be in two sections at the same time
2. A section cannot have two subjects in the same slot
3. A room cannot have two classes at the same time
4. Teacher must not exceed max periods per day (configurable)

**Algorithm:**
- Use constraint validation on save, not auto-generation
- Visual drag-and-drop grid (day × period)
- Color-code by subject
- Print-ready timetable per class, per teacher

**Substitution Management:**
- When teacher is absent → assign substitute
- Substitution log maintained
- Auto-notify substitute teacher via SMS

---

### MODULE 4 — Attendance System (Very Deep)

**Student Attendance:**
- Marked per section per day
- Three methods: Manual (web), QR code scan, Biometric device sync
- Late arrival tracking (separate from absent)
- Half-day leave handling
- Holiday calendar integration (marked days auto-set to Holiday)
- Monthly sheet view (calendar grid)
- Auto-SMS to parent when child is absent (fire via queue, 15 min after school start)

**Staff Attendance:**
- Check-in / check-out times
- Biometric device API integration (ZKTeco — you already have experience here!)
- Late deduction rules (configurable grace period)
- Link to payroll — attendance affects salary

**Reports:**
- Daily defaulters list (for admin)
- Monthly attendance % per student
- Students below threshold (e.g., below 75%) → auto-flag
- Staff attendance summary for payroll

---

### MODULE 5 — Fee Management (Deepest Module)

This is the most critical and most customized module for Pakistani schools.

**Fee Generation Flow:**
```
Define Fee Structure per class → Assign to students → 
Auto-generate monthly invoices (via Scheduler) → 
Generate Challan → Parent pays → Record payment → 
Update status → SMS receipt
```

**Fee Structure Setup:**
- Per class, per fee type, per academic year
- Effective date support (fee can change mid-year)
- Copy previous year structure with override

**Pakistani Challan Format:**
- School copy + Bank copy + Parent copy (triplicate)
- Bank code, account number (HBL/BOP/Meezan format)
- Due date printed
- Challan number as unique reference
- QR code optional for payment verification

**Discount Engine:**
- Multiple discounts stackable (sibling + scholarship)
- Cap at 100% discount
- Approval workflow for manual discounts
- Time-bound discounts (scholarship for this year only)

**Fine Calculation:**
- Configurable: PKR X per day after due date
- Grace period (first 5 days free)
- Fine waiver request + approval flow
- Monthly fine report for accountant

**Sibling Discount Logic:**
```
If 2 siblings enrolled:
  eldest = 0% discount
  2nd child = 10% on monthly fee
  3rd child = 20% on monthly fee
  (configurable per school)
```

**Fee Defaulters Workflow:**
1. Auto-flag after due date passes
2. SMS reminder sent (day 1, day 7, day 15)
3. Defaulters list available daily
4. Option to restrict exam admission if fee unpaid
5. TC (Transfer Certificate) hold if dues pending

---

### MODULE 6 — Examination & Results

**Exam Setup:**
- Exam types with weightage (e.g., Mid = 30%, Final = 70%)
- Per-subject total marks and passing marks
- Schedule with room allocation
- Seating plan generation

**Marks Entry:**
- Per subject per student
- Bulk entry via Excel upload (teacher uploads sheet)
- Absent marking (no marks entered = 0 if not marked absent)
- Lock marks after verification (prevent accidental edits)
- Re-check request workflow

**Result Calculation:**
```
Weighted Result = (Mid Marks / Mid Total × Mid Weightage)
               + (Final Marks / Final Total × Final Weightage)
Final Percentage = Weighted Result × 100
Grade = lookup grading_scale table
Position = rank within section / class
```

**Result Card Generation:**
- School header with logo
- Student photo, class, section, roll no
- Subject-wise marks table
- Total, percentage, grade, position
- Teacher/principal signature line
- Print-ready PDF (per student or bulk)

**Analytics:**
- Subject-wise average per class
- Top 10 position holders
- Fail rate per subject
- Comparison across terms

## ⚠️ PART 7 — CRITICAL EDGE CASES & LOOPHOLES

These are things most developers miss and regret later:

**Academic Year Transition:**
- What happens to unpaid fees when year ends? → Carry forward as arrears
- Promoted student gets new fee structure automatically
- Old exam records must remain accessible after year closes

**Mid-Year Admission:**
- Fee for partial month — pro-rate or full month? (make it configurable)
- Which exams has the student missed? — partial result handling

**Transfer Certificate (TC):**
- System must block TC if fee dues or library book pending
- TC number must be unique and sequential
- TC issued = student status → Left

**Duplicate Prevention:**
- Same B-Form number in same campus = hard block
- Same teacher double-booked in timetable = hard block
- Same student marked attendance twice = soft warning

**Data Deletion Policy:**
- Never hard-delete students, employees, or financial records
- Use `soft deletes` everywhere
- Audit log on every critical action

**Academic Year Lock:**
- Once year is closed, marks/fees/attendance become read-only
- Only super admin can unlock with reason logged

**SMS Failures:**
- Failed SMS must be retried (queue with retry policy)
- Parent phone number validation (Pakistani format: 03XX-XXXXXXX)
- Log every SMS attempt regardless of success

**Fee Edge Cases:**
- Student leaves mid-month — fee waived or charged? (policy setting)
- New fee type added mid-year — apply retroactively or from next month?
- Bank returns a payment — reverse the receipt cleanly

**Multi-Campus:**
- Student transfers between campuses = preserve full history
- Report at group level (all campuses combined)
- Each campus has its own fee structure (same group, different pricing)
